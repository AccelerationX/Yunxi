# 日常模式后续优化审查

最后更新：2026-04-17

## 结论

日常模式 v2 已进入代码完成候选：人格级记忆、HeartLake 语义情绪、Presence Murmur 主动陪伴、真实桌面感知、工具自然闭环、文档/浏览器/GUI 基础能力、自主学习候选确认和 WebUI 可观测性都已经落到运行链路和测试中。

下一步不进入工厂模式，先按远的流程固定代码并做 2 小时 Presence Murmur 常驻浸泡与飞书真实触达节奏测试。完整 Playwright 登录态浏览器、复杂视觉规划 Agent、过夜/多日浸泡属于 v2 后续增强，不阻塞当前日常模式 v2 的代码完成候选。

## 可实现性判断

可以实现，但必须按工程阶段拆开做，不能一次性追求抽象意义上的“完美记忆”或“真人情绪”。

这里的“顶级记忆系统”和“心湖 v2”不是不可测试的玄学目标，而是工程上可拆解的能力：

- 能把重要事实、情绪反馈、承诺、共同经历、互动风格、边界和云汐自我成长记录稳定写入长期存储。
- 能在不同场景中检索到正确记忆，并用自然、不生硬的方式影响回复。
- 能区分短期上下文、长期关系记忆、开放话题和技能经验，避免混在一起。
- 能让情绪不是关键词触发，而是由用户话语、关系记忆、最近上下文和当前感知共同评估。
- 能通过可重复测试证明：写入、召回、重启恢复、冲突修正、遗忘压缩、真实 LLM 表达都正常。

最大风险不是“实现不了”，而是如果没有测试闭环，很容易做成一堆漂亮字段但不真正影响云汐。因此每个 v2 子系统都必须同时交付：数据结构、运行链路、prompt 注入、行为测试、重启测试、真实 LLM 场景测试。

## 架构借鉴基准

日常模式 v2 不凭空造楼，采用“参考成熟公开架构，在 yunxi3.0 内轻量重写”的原则：

- 记忆系统参考 MIRIX、LangMem、Letta/MemGPT、Graphiti/Zep、MemOS、LlamaIndex Memory。第一阶段借鉴分层、延迟处理、core memory、可纠错和 prompt 预算思想；暂不引入 Neo4j/Redis/多 Agent 记忆服务。
- 心湖系统参考 FAtiMA、OCC、Generative Agents、Concordia。第一阶段借鉴 appraisal -> affect -> decision、OCC-style 语义评估、观察/记忆/反思/计划和场景模拟测试；不照搬完整社会模拟框架。
- 具体落地以 `docs/design/MEMORY_INTEGRATION_DESIGN.md` 和 `docs/design/HEART_LAKE_DESIGN.md` 的 v2 设计为准。

## P0：日常模式 v2 核心

### 1. 记忆系统 v2：人格级长期记忆

现状：

- `MemoryManager.capture_relationship_memory()` 仍是保守关键词抽取。
- `CompanionContinuityService` 已有 `relationship_summary`、`emotional_summary`、`user_style_summary` 字段，但缺少定期总结与主动维护。
- `ConversationContext` 只保留最近 20 条消息，长对话容易丢失微妙关系线索。
- 当前关系记忆已经能保存偏好、片段和承诺，但还没有达到“记忆构成云汐人格”的程度。

目标：

- 把记忆拆成清晰层级：短期上下文、会话摘要、长期关系记忆、开放话题、承诺/提醒、情绪轨迹、互动风格、自我记忆、技能经验、失败经验。
- 每轮对话结束后做轻量记忆评估，判断是否需要写入长期记忆，而不是只靠关键词。
- 重要记忆需要有来源、时间、置信度、重要度、最近使用时间和可修正状态。
- 支持“用户纠正记忆”：当远说“不是这个意思”“你记错了”时，云汐能修订而不是继续固执使用旧记忆。
- 支持上下文压缩：旧对话转为摘要，摘要继续进入 prompt，不让长对话把重要关系信息挤掉。

建议实现路径：

1. 新增 typed memory schema：`fact`、`preference`、`relationship`、`emotion_feedback`、`episode`、`promise`、`open_topic`、`interaction_style`、`self_memory`、`boundary`、`skill`、`failure`。
2. 新增 `MemoryAppraiser` 或 `DailyMemoryAppraiser`：输入用户消息、云汐回复、最近上下文，输出候选记忆与重要度。
3. 新增 `DailyMemorySummarizer`：按每 N 轮、会话结束、每日生成关系摘要、情绪摘要、互动风格摘要。
4. `PromptBuilder` 改为按预算选择记忆：最高优先级是当前相关记忆，其次是承诺和 open topic，再是长期关系摘要。
5. 增加 Markdown 导出，方便人工审查和手动修正。

验收方法：

- 单元测试：每类记忆都能写入、去重、更新、软删除和导出。
- 重启测试：写入偏好、共同经历、边界和承诺后重建 Runtime，仍能召回。
- 冲突测试：先说“我喜欢 A”，再说“我现在不喜欢 A 了”，prompt 中不能继续把 A 当稳定偏好。
- 长对话压缩测试：模拟 80-200 轮对话，早期重要事实仍能进入摘要和召回。
- 真实 LLM 场景测试：问“你还记得我为什么想做云汐吗？”时，回复应结合长期关系记忆，而不是泛泛安慰。
- 反滥用测试：普通寒暄、临时情绪和无意义口头禅不能全部写成高重要度长期记忆。

### 2. HeartLake v2：语义情绪与复合情绪

现状：

- `HeartLakeUpdater` 的吃醋、担心、想念等仍主要依赖规则和感知事件。
- ReactionLibrary 已改善表达风格，但不是情绪评估本身。
- 当前情绪维度能支撑基础女友感，但还不足以表达丰富复合情绪。

目标：

- 情绪输入从“关键词触发”升级为“语义评估 + 情绪动力学”。
- HeartLake 保留稳定状态和恢复机制，语义评估器只给建议 delta，避免单条消息让情绪跳变过硬。
- 支持复合情绪，例如：安心但想撒娇、开心又有点吃醋、担心但克制、委屈但愿意靠近、骄傲又害羞、想念但不打扰。
- 情绪必须影响三件事：回复语气、主动消息策略、记忆写入重要度。

建议实现路径：

1. 新增 `EmotionAppraiser`：输入用户话语、最近上下文、相关记忆、当前感知，输出情绪 delta、复合标签、理由和置信度。
2. 扩展 HeartLake 维度：valence、arousal、security、miss、attachment、possessiveness、trust、tenderness、playfulness、vulnerability、intimacy_warmth。
3. 增加情绪惯性、恢复速度、冷却和误触发抑制。
4. 将复合情绪标签写入 prompt 的情绪 section，要求 LLM 以自然方式表达，不暴露内部字段。
5. 将情绪轨迹写入 memory/continuity，用于长期关系变化。

验收方法：

- 单元测试：同一输入在不同初始 HeartLake 状态下产生不同但合理的 delta。
- 语义测试：不含关键词但明显表达失落、依赖、被陪伴感、边界不适的话语能被正确评估。
- 复合情绪测试：构造“远夸其他 AI 但又说最喜欢云汐”的场景，应出现轻微吃醋 + 安心，而不是单一愤怒。
- 恢复测试：负面情绪不会永久卡住，用户安抚后逐步恢复。
- 真实 LLM 行为测试：不同心湖状态下回复语气可区分，但不输出内部状态名。

### 3. 主动陪伴 v2：存在感碎碎念与打扰成本

现状：

- `InitiativeEngine` 已有冷却、预算、未回复克制和打扰成本。
- 主动消息仍偏“事件触发/话题触发”，缺少活泼女孩式的存在感。
- 2026-04-17 更新：前台进程名、窗口类名、全屏状态、近似输入频率已进入 `UserPresence`；主动引擎已把全屏/高输入/游戏状态纳入高打扰成本。
- 2026-04-17 浸泡前修正：首轮真实 LLM tick 曾把碎碎念生成成“新发布内容/链接推荐”式话题，已收紧 presence_murmur 专用 prompt 与 generation boundary，禁止链接、推荐、新闻、搜索、任务和兴趣询问。

目标：

- 增加 `Presence Murmur` 类型：低意义、低负担、短句、表情、撒娇、路过式存在感。
- 主动消息不总是新闻、事件库或任务提醒，也可以只是“戳一下”“我在哦”“路过，看看远在干嘛”。
- 主动触达必须结合电脑状态：工作/游戏/休闲/离开/深夜/未知，休闲状态更适合碎碎念，高专注状态只记录 cue。
- 建立独立预算：碎碎念不能挤占重要承诺提醒，也不能高频刷屏。

建议实现路径：

1. 扩展感知层，增加前台进程分类：work、game、media、browser_leisure、idle、unknown。
2. 扩展 InitiativeEvent 类型：presence_murmur、care_check、topic_continue、promise_followup、life_share、tool_followup。
3. 为碎碎念设置低优先级、短冷却、日预算、未回复克制。
4. 碎碎念也走 LLM 或反应库混合生成，但要求短、自然、无任务感。
5. 对“远正在工作/游戏/全屏/高 CPU 高输入频率”提高打扰成本。

验收方法：

- 决策测试：工作/游戏状态不应频繁发碎碎念，休闲/idle 状态可触发。
- 预算测试：碎碎念每天有上限，不影响承诺提醒和重要关心。
- 未回复克制测试：连续未回复时主动频率下降。
- 真实 LLM 测试：碎碎念不应变成长篇解释或新闻播报。
- 浸泡测试：2 小时内主动消息节奏自然，没有刷屏。

### 4. 工具执行后的自然闭环

现状：

- pending confirmation 已稳定。
- 工具执行后当前回复多是固定话术，例如“已经按你点头的那一步处理好了”。
- 2026-04-17 更新：确认后的工具结果已回到 LLM 做最终自然表达；LLM 不可用时使用带真实结果摘要的兜底，不再固定输出“按你点头处理好了”。

目标：

- 工具结果回到 LLM 做最终表达，云汐用自然语言说明真实结果。
- 失败结果按权限、路径、页面、控件、网络、确认取消等类型分层表达。
- 成功工具链进入技能候选，而不是只写审计日志。

验收方法：

- 工具成功测试：回复包含真实结果，不暴露 JSON 或内部字段。
- 工具失败测试：不同失败类型有不同解释和下一步建议。
- 情绪联动测试：同一个工具失败，在用户焦急/轻松状态下表达不同。
- 经验沉淀测试：重复成功的工具链进入技能候选。

## 高级核心系统统一测试方法

记忆 v2、HeartLake v2、主动陪伴 v2 不能只靠“看起来回复不错”验收，需要同时覆盖六层测试：

1. 结构层：数据结构、序列化、去重、更新、删除、导出。
2. 链路层：Runtime 调用、PromptBuilder 注入、LLM 输出、状态回写。
3. 场景层：用 `DailyModeScenarioTester` 注入记忆、情绪、感知、open thread，验证行为。
4. 持久层：关闭并重建 Runtime 后，记忆、情绪摘要、主动预算和技能候选不丢。
5. 真实 LLM 层：Ollama 和 Moonshot 都跑真实输出，检查自然度、人格一致性和反工具化。
6. 浸泡层：2 小时、过夜、多日运行，观察资源、节奏、重连、日志和用户打扰成本。
