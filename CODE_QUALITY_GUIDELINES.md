# 云汐 3.0 代码实现准则

> **效力等级**：最高约束。  
> **适用范围**：所有写入 `yunxi3.0` 代码库的修改。  
> **核心信条**：代码质量优先，实现速度次要，最优效果是唯一目标。

---

## 一、总纲

1. **质量永远优先于速度**
   - 宁可一个模块多花三天打磨，也不写"先跑起来再说"的临时代码。
   - 不允许以"赶进度"为由降低接口设计标准、绕过异常处理、省略必要测试。

2. **最优效果是唯一验收标准**
   - 如果当前实现只能达到 70% 的效果，那么它等于 0%。
   - 每个功能必须在真实场景下验证其效果，测试通过方可视为完成。

3. **拒绝"以后再说"的技术债务**
   - 不写 TODO 占位代码（文档中的规划性 TODO 除外）。
   - 不引入"临时兼容层"。
   - 不保留"也许以后会用到"的死代码。

4. **所有代码必须在 yunxi3.0 内完整实现**
   - 不允许跨目录 import `D:\ResearchProjects` 下的任何文件。
   - 研究成果只能作为"思路参考"，不能作为"代码依赖"。

5. **诚实报告问题，测试失败必须立即解决**
   - 代码编写过程中出现的任何问题（编译错误、测试失败、设计冲突、依赖缺失）必须如实记录并立即解决，禁止一带而过或隐藏问题。
   - 测试未通过等同于功能未实现，不得以"先往后做"为由跳过修复。

---

## 二、接口设计准则

### 2.1 接口即契约

- 每个公共类、公共方法必须有**类型注解**和**docstring**。
- 返回值类型必须明确，禁止返回 `Any` 或裸 `dict` 作为主要接口的返回值。
- 异常必须在 docstring 中声明：可能抛出什么异常，在什么场景下抛出。

**示例**：
```python
from typing import Optional
from dataclasses import dataclass

@dataclass
class ToolResult:
    """工具执行的标准化返回类型"""
    success: bool
    output: str
    error: Optional[str] = None

class UIADriver:
    """基于 Windows UI Automation 的桌面操作驱动器。"""
    
    def find_window_by_title(self, keyword: str) -> Optional[auto.WindowControl]:
        """
        按标题关键词查找顶层窗口。
        
        Args:
            keyword: 窗口标题包含的关键词，大小写不敏感。
        
        Returns:
            匹配到的 WindowControl，未找到时返回 None。
        """
```

### 2.2 单一职责原则

- 一个类只做一件事。
- 如果类的 docstring 里出现了"和"、"同时负责"等词，说明职责过重，必须拆分。
- 禁止出现 2.0 中 `CompanionRuntimeSupport` 那样的上帝类。

### 2.3 依赖注入优先于全局状态

- 所有外部依赖（LLM provider、MCP Hub、MemoryManager）必须通过构造函数注入。
- 禁止在模块级别创建全局单例实例。
- 禁止在方法内部 import 并直接使用未注入的模块级变量。

## 三、实现准则

### 3.1 防御式编程

- **所有外部输入必须校验**：用户输入、LLM 输出、文件内容、网络响应、环境变量。
- **所有 IO 操作必须有超时**：子进程、网络请求、文件锁、数据库连接。
- **所有资源必须有清理保证**：`try/finally`、`async with`、上下文管理器。

```python
# 正确示例
async def call_with_timeout(coro, timeout: float = 30.0):
    try:
        return await asyncio.wait_for(coro, timeout=timeout)
    except asyncio.TimeoutError:
        logger.error("操作超时")
        raise
    finally:
        # 资源清理
        pass
```

### 3.2 错误处理

- 异常必须被**捕获、记录、转化**为对用户有意义的错误信息。
- 不允许裸 `except:` 或 `except Exception:` 后不记录日志。
- 底层错误不允许直接冒泡到最上层，每层应该添加自己的上下文信息。

```python
# 错误示例
except Exception as e:
    raise  # 没有价值

# 正确示例
except httpx.ConnectError as e:
    logger.error(f"MCP Server 连接失败: {e}")
    raise MCPServerUnavailableError(f"无法连接到 {server_name}") from e
```

### 3.3 代码可读性

- 变量名和函数名必须"自解释"。
  - 好：`calculate_dependency_depth`
  - 坏：`calc_dep_d`
- 禁止使用数字常量魔法值，必须定义为有名字的常量。
- 嵌套深度不超过 4 层；超过时必须拆分为辅助函数。

### 3.4 异步规范

- IO 操作必须走 `async`/`await`。
- 阻塞式库（如 `sentence-transformers`）必须在线程池中执行：`await asyncio.to_thread(...)`。
- 禁止 sync/async 混用。如果一个方法涉及 IO，它必须是 `async`。

### 3.5 测试即代码的一部分

- 每个公共方法必须有其对应的测试或调用路径的端到端验证。
- 测试不是"写完功能后补"的，而是"与功能同步写"的。
- 所有 bug 修复必须附带回归测试。

## 四、重构准则

### 4.1 重构是常态

- 发现代码有坏味道时，**立即重构**，不要等"下一个版本"。
- 重构前必须确保当前测试通过。
- 重构后必须再次运行全量测试。

### 4.2 删除代码的勇气

- 如果一段代码的调用者为 0，立即删除。
- 如果兼容层只是为了"以防万一"，立即删除。
- 如果 V1 和 V2 并存且 V1 已无价值，立即删除 V1。

### 4.3 最小侵入原则

- 修改现有代码时，只改动必要的部分。
- 不因为"顺便"而修改无关代码。
- 一个 commit 只解决一个问题。

---

## 五、文档与注释准则

### 5.1 代码即文档

- 好的代码不需要过多注释来解释"做了什么"。
- 注释应该解释"为什么这么做"，而不是"做了什么"。

```python
# 错误注释
# 遍历列表中的每个元素
for item in items:
    ...

# 正确注释
# 使用倒序遍历，避免删除元素时索引错位
for item in reversed(items):
    ...
```

### 5.2 设计文档同步

- 任何架构变更、接口变更、新增模块，必须在对应的 `docs/design/*.md` 中更新。
- 设计文档中的接口定义必须与代码实现保持一致。
- 已实现的功能在设计文档中标记为 `✅ 已完成`，未实现的标记为 `⏳ 待实现`。

## 六、禁止清单

以下行为在云汐 3.0 的开发中**严格禁止**：

| 序号 | 禁止行为 | 正确做法 |
|------|---------|---------|
| 1 | `import D:/ResearchProjects/...` | 在 yunxi3.0 内重写 |
| 2 | 返回 `Any` 或裸 `dict` 作为主要接口返回值 | 定义 `dataclass` 或 TypedDict |
| 3 | 裸 `except:` 后 `pass` | 捕获具体异常并记录/转化 |
| 4 | sync/async 混用（如 async 方法调用 sync API 不转线程） | 统一 async，阻塞调用包 `to_thread` |
| 5 | 无超时的 IO 操作 | 所有 IO 必须带 timeout |
| 6 | 未经验证的假设（如"Ollama 一定在 localhost:11434"） | 显式检查可用性并降级 |
| 7 | 为测试通过而降低安全标准 | 修改测试以符合正确逻辑 |
| 8 | 写 TODO 占位代码 | 不实现则不要写，文档中记录即可 |
| 9 | 一个函数超过 80 行（不含 docstring） | 拆分为多个函数 |
| 10 | 魔法数字硬编码 | 定义为命名常量 |

## 七、Commit 与版本控制准则

1. **原子性 Commit**
   - 一个 commit 只解决一个逻辑问题。
   - 不允许"修复 bug + 重构 + 新增功能"混在一起提交。

2. **清晰的 Commit Message**
   ```
   [模块] 动作: 简短描述
   
   - 详细说明 1
   - 详细说明 2
   ```
   示例：
   ```
   [mcp] feat: 实现 MCPHub 的初始化与工具发现
   
   - 封装官方 mcp.Client 为 stdio 模式
   - 支持多 Server 并发初始化
   - 添加初始化失败时的降级策略
   ```

3. **每次 commit 前必须通过本地测试**
   - 至少运行相关模块的 pytest。
   - 如果有接口变更，运行 `ConversationTester` 相关用例。

---

## 九、云汐的人格与产品准则

1. **云汐不是死板的脚本执行程序**
   - 每一行代码都应服务于"让云汐表现得像一个有主动性、有情感、能判断情境的伙伴"，而非一个只会按固定步骤执行的脚本引擎。
   - 禁止将实现简化为"if-else 罗列所有情况然后按流程走一遍"；必须保留情境感知、意图推断、策略选择、情感反馈等层次。
   - 接口和类的命名、返回值的设计、错误信息的措辞，都应体现人格化特征，而不是冷冰冰的系统日志风格。

---

## 八、总结

> **写代码不是为了"让它跑起来"，而是为了"让它在真实世界里稳定、优雅、可维护地创造价值"。**

云汐 3.0 不是一次性的校园项目demo，而是一个具备工程严谨性和产品完整度的系统。每一行代码都应该经得起以下三个问题的拷问：

1. **它是否足够可靠？** 在真实用户、真实桌面、真实网络环境下不会崩溃。
2. **它是否足够清晰？** 三个月后的我或其他人能一眼看懂它的意图。
3. **它是否足够完整？** 不只是一个函数的堆砌，而是真正连接到了用户可见的体验中。

如果答案有一个是"否"，那就重写，直到三个都是"是"。
